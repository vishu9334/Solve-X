Mentor Verification System

Internal Technical Document | Week 1 Sprint | Version 1.0 | June 2026

1. System Overview

This document describes the complete architecture of the Mentor Verification System — covering Authentication, Mentor Registration, Assessment Engine, Real-time Activity Monitoring, AI Evaluation Agent, and Email Notification Pipeline.

The system supports three types of users: Students, Mentors, and Admins. A mentor must pass a proctored assessment before gaining verified status on the platform. Admins create and manage assessments, oversee mentor verification, and control platform-level operations. This document is the single source of truth for Week 1 development.

2. High-Level Architecture (HLA)

The platform is divided into five major domains. Each domain is independently owned by a team member this week. All domains share the MongoDB cluster and communicate through well-defined API contracts.

2.1 Domain Map

2.2 High-Level Request Flow

  1. Admin registers/logs in and creates an AssessmentStore document for a given category (DSA, MERN, etc.) with questions, duration, and passing percentage.
  1. Mentor visits platform and chooses Simple Auth or Google OAuth to register.
  1. On successful registration, role='mentor' triggers MentorProfile creation with isVerifiedMentor: false and verificationStatus: 'pending'.
  1. A 3-day cron monitors all pending mentors. If no assessment attempt is made within 72 hours, the account and profile are deleted automatically.
  1. Mentor clicks "Start Assessment". Assessment Engine creates an AssessmentActivitySession and returns the timer config from the Admin-created AssessmentStore.
  1. While the assessment runs, the Activity Monitor streams events (tab switch, screen resize, etc.) from the browser via SSE into a Redis stream. The backend consumer persists every event to the session document.
  1. Assessment ends (submit, timeout, or cancel). Member 2's engine saves the score. Member 3's AI agent then reads the full session — score + activity events — and sets activityDecision and verificationStatus.
  1. Email is dispatched on both pass and fail outcomes.
  1. On fail: Attempt model records the attempt. A 7-day retry lock is enforced. After a second failure, another 7-day lock. If 1 month passes without clearance, the account is deleted and the email is blacklisted.
  1. Admin can view all mentor verification statuses, manually manage blacklist, and monitor platform activity via admin-only routes.

3. Low-Level Architecture — Auth & User Layer (Member 1)

3.1 Schema Design

The user identity system uses Mongoose discriminators. All users share a single CommonUser collection. The authType field acts as the discriminator key, routing to either SimpleUserAuth or GoogleAuth sub-documents.

3.1.1 CommonUser (base schema)

FieldTypeDefaultNotes
nameStringrequiredTrimmed
emailStringrequired, uniqueLowercase, indexed
isVerifiedUserBooleanfalseEmail verification flag
avatarStringnullProfile image URL
roleEnum'student'admin / mentor / student
authTypeStringautoDiscriminator key: SimpleUserAuth or GoogleAuth

3.1.2 SimpleUserAuth (discriminator)

  • Extends CommonUser with a hashed password field.
  • bcrypt pre-save hook hashes password at salt 10.
  • comparePassword() instance method for login validation.
  • toJSON() strips password from all API responses.

3.1.3 GoogleAuth (discriminator)

  • Extends CommonUser with googleId (indexed, sparse), accessToken, refreshToken, idToken.
  • passport-google-oauth2 strategy populates these on callback.

3.1.4 AdminProfile

FieldTypeDefaultNotes
userIdObjectIdrequiredRef: CommonUser
bioString—Admin's bio / description
createdAt / updatedAtDateautoMongoose timestamps
Admin profile is intentionally minimal. Admin permissions are derived from role = 'admin' in CommonUser and enforced via middleware — not from profile fields.

3.2 API Endpoints — Auth

MethodRouteAuthDescription

POST/auth/loginNoneEmail + password login; returns JWT
GET/auth/googleNoneRedirect to Google OAuth consent screen
GET/auth/google/callbackNoneOAuth callback; upsert GoogleAuth user; return JWT

3.3 Registration Logic (Step by Step)

  1. Validate request body — name, email, password (min 8 chars), role.
  1. Check if email already exists in CommonUser. If yes, return 409 Conflict.
  1. Check email against Blacklist collection. If found, return 403 Forbidden.
  1. Create SimpleUserAuth document. Pre-save hook hashes password.
  1. Based on role: create MentorProfile (isVerifiedMentor: false, verificationStatus: 'pending') OR StudentProfile OR AdminProfile.
  1. Issue JWT (payload: userId, role, email). Return 201 with user object (password stripped via toJSON).

3.4 Cron Jobs

3.4.1 3-Day Mentor Cleanup (runs daily at midnight)

  • Query: MentorProfile where verificationStatus = 'pending' AND createdAt <= now - 72 hours.
  • For each result: check Attempt model for any attempt linked to userId.
  • If no attempt found: delete MentorProfile, delete CommonUser document.
  • Log deleted accounts for audit trail.

3.4.2 1-Month Blacklist Cleanup (runs daily)

  • Query: MentorProfile where verificationStatus = 'rejected' AND rejectedAt <= now - 30 days.
  • Fetch email from CommonUser. Add to Blacklist collection (email, blacklistedAt, reason).
  • Delete MentorProfile and CommonUser.
  • On any future register attempt with blacklisted email: return 403 with message 'Account permanently restricted'.

3.5 Low-Level Architecture — Admin Layer (Member 1)

3.5.1 Admin Role & Access

Admin is the platform operator. The role = 'admin' field in CommonUser is the single source of admin identity. All admin routes are protected by a requireAdmin middleware that checks req.user.role === 'admin' from the decoded JWT.

Admin has no assessment flow — admins do not take assessments. Their responsibilities are:

  • Create and manage AssessmentStore documents (the test content).
  • View all mentor profiles and their verification status.
  • Manually manage the email blacklist.
  • Monitor platform-level activity.

3.5.2 Admin API Endpoints

MethodRouteAuthDescription
POST/admin/assessment/createJWT (admin)Create a new AssessmentStore for a category
GET/admin/assessment/listJWT (admin)List all assessments with category and stats
PATCH/admin/assessment/:idJWT (admin)Update assessment — questions, duration, passingPercentage
DELETE/admin/assessment/:idJWT (admin)Delete an assessment
GET/admin/mentorsJWT (admin)List all mentors with verificationStatus
GET/admin/mentors/:idJWT (admin)Get single mentor profile + attempt history
GET/admin/blacklistJWT (admin)View all blacklisted emails
DELETE/admin/blacklist/:emailJWT (admin)Remove email from blacklist (unban)

3.5.3 Assessment Creation — Detailed Logic

  1. Admin logs in → JWT issued with role: 'admin'.
  1. POST /admin/assessment/create with body: { title, category, durationMinutes, totalQuestions, passingPercentage, maxWarningsAllowed }.
  1. Middleware requireAdmin validates role === 'admin' from JWT. Reject with 403 if not admin.
  1. Validate all required fields. passingPercentage must be between 1–100. durationMinutes must be > 0.
  1. Set createdBy: req.user.userId — this links the assessment to the admin who created it.
  1. Save AssessmentStore document. Return 201 with created assessment.
Note: AssessmentStore.createdBy references CommonUser — this is why admin must be a CommonUser first. The admin's userId from CommonUser is what goes into createdBy.

3.5.4 Mentor Oversight

  • GET /admin/mentors returns all MentorProfile documents with verificationStatus, isVerifiedMentor, skillCategory, verifiedAt, rejectedAt, rejectionReason.
  • Admin can view mentor attempt history by fetching Attempt model for a given userId.
  • Admin cannot manually approve a mentor — verification is purely AI-driven. This keeps the process objective and tamper-proof.

3.5.5 Blacklist Management

  • Admin can view the full Blacklist collection via GET /admin/blacklist.
  • Admin can unban an email via DELETE /admin/blacklist/:email — this removes the email from the blacklist, allowing re-registration.
  • All unban actions should be logged with adminId, email, and timestamp for audit purposes.

3.5.6 requireAdmin Middleware

// Pseudo-code
export const requireAdmin = (req, res, next) => {
  const { role } = req.user; // decoded from JWT
  if (role !== 'admin') return res.status(403).json({ message: 'Admin access required' });
  next();
};

4.1 Schema Design

4.1.2 AssessmentActivitySession

This is the core session document. Every assessment run — whether submitted, cancelled, or timed out — creates one document.

FieldTypeDefaultNotes

startedAtDaterequiredSet on POST /start

4.1.3 Attempt Model

  • userId: ObjectId (ref CommonUser) — one Attempt document per mentor.
  • attempts[]: array of AssessmentActivitySession ObjectIds.
  • createdAt / updatedAt from timestamps — used to calculate retry lock windows.

4.2 Assessment Lifecycle — API Endpoints

MethodRouteAuthDescription
POST/assessment/startJWT (mentor)Create session, log TEST_STARTED, return timer
PATCH/assessment/saveJWT (mentor)Auto-save current answers
POST/assessment/submitJWT (mentor)Final submit — calculate score, trigger AI
POST/assessment/cancelJWT (mentor)Cancel — auto-save state, log PAGE_CLOSE

4.3 Assessment Start — Detailed Logic

  1. Authenticate JWT. Extract userId, role. Ensure role = 'mentor'.
  1. Check Attempt model for userId. If attempts.length >= 2, return 403 'Maximum attempts reached'.
  1. Check last attempt's updatedAt. If within 7 days, return 403 'Retry locked until <date>'.
  1. Fetch AssessmentStore by category (from request body).
  1. Create AssessmentActivitySession: set userId, category, startedAt: Date.now().
  1. Push TEST_STARTED event into events[] with severity 'info'.
  1. Update Attempt.attempts[] with new session ID.
  1. Return: sessionId, durationMinutes, totalQuestions.

4.4 Auto-Save + Timer Logic

  • Frontend sends PATCH /assessment/save every 30 seconds with current answers.
  • Backend overwrites saved answers in the session document (no versioning needed).
  • Frontend timer counts down from durationMinutes. On expiry: POST /assessment/submit is called automatically.
  • Backend also validates: if (Date.now() - startedAt) > durationMinutes * 60 * 1000, reject late submits.
  • On expiry: push TIME_EXPIRED event (severity: critical). Set endedAt. Calculate score.

4.5 Cancel / Mid-Exit Handling

  • If mentor clicks Cancel or closes the browser tab, frontend sends POST /assessment/cancel.
  • Backend: push PAGE_CLOSE event (severity: critical). Set endedAt = Date.now(). Save current state.
  • The attempt is still counted. Retry lock applies.
  • hasPageClose flag is set to true — AI agent reads this in its decision.

4.6 Result Calculation & Save

  1. Count correct answers. Score % = (correct / totalQuestions) * 100.
  1. Compare with AssessmentStore.passingPercentage.
  1. Update MentorProfile:
    • Pass: verificationStatus = 'approved', isVerifiedMentor = true, verifiedAt = Date.now().
    • Fail: verificationStatus = 'rejected', rejectedAt = Date.now(), rejectionReason = <reason string>.
  1. Update lastAssessmentAttemptId in MentorProfile.
  1. Emit internal event to trigger Member 3's AI agent and email pipeline.

4.7 Retry Logic State Machine

StateConditionAction
First attempt — passscore >= passingPercentageApproved. isVerifiedMentor = true
First attempt — failscore < passingPercentageRejected. 7-day retry lock starts
Second attempt — passscore >= passingPercentageApproved. isVerifiedMentor = true
Second attempt — failscore < passingPercentageRejected. 7-day lock. No more retries
1 month no clearancerejectedAt <= now - 30 days AND attempts >= 2Account deleted. Email blacklisted

5. Low-Level Architecture — Activity Monitor (Member 3)

5.1 SSE + Redis Pipeline

During the assessment, the browser maintains a persistent SSE connection to the backend. All suspicious browser events are captured by the frontend and sent via a separate HTTP POST endpoint. The backend then publishes each event to a Redis stream. A separate Redis consumer reads from the stream and persists events to MongoDB.

5.2 Frontend Events Captured

Event TypeSeverityTrigger Condition
TEST_STARTEDinfoAssessment start button clicked
FULLSCREEN_ENTEREDinfoBrowser goes fullscreen
FULLSCREEN_EXITEDwarningFullscreen exited during test
TAB_SWITCHEDcriticaldocument.visibilityState = hidden
TAB_RETURNEDinfodocument.visibilityState = visible again
WINDOW_BLURwarningwindow.onblur event fired
WINDOW_FOCUSinfowindow.onfocus event fired
SCREEN_RESIZEDwarningwindow.innerWidth / innerHeight changed
PAGE_REFRESHcriticalbeforeunload with reload detected
PAGE_CLOSEcriticalbeforeunload / cancel triggered
TIME_EXPIREDcriticalTimer countdown reaches zero
TEST_SUBMITTEDinfoMentor manually submits

5.3 Screen Snapshot (sent with every event)

Every event payload includes a screenSnapshotSchema object:

  • innerWidth, innerHeight — browser viewport dimensions.
  • outerWidth, outerHeight — full browser window dimensions.
  • screenWidth, screenHeight — physical display resolution.
  • availWidth, availHeight — usable screen area.
  • isFullscreen — true if browser is in fullscreen mode.
  • isTabHidden — true if tab is not visible.
  • hasFocus — true if browser window has focus.

5.4 Event Ingestion — Backend Flow

  1. Frontend POST /assessment/activity/event with { sessionId, eventType, screen, metadata }.
  1. Backend validates sessionId and JWT. Publishes to Redis stream key: assessment:activity:<sessionId>.
  1. Redis consumer (separate worker process) reads from stream.
  1. For each event: push into AssessmentActivitySession.events[], increment totalEvents.
  1. If severity = 'warning': increment warningCount.
  1. If severity = 'critical': increment criticalCount.
  1. Set convenience boolean flags: hasTabSwitch, hasFullscreenExit, hasScreenResize, hasPageClose based on eventType.
  1. Save document. Redis ACK the message.

6. Low-Level Architecture — AI Evaluation Agent (Member 3)

6.1 When AI Agent Runs

The AI agent is triggered automatically after POST /assessment/submit or POST /assessment/cancel is processed by Member 2's engine. It runs as an async background job — it does not block the API response.

6.2 Behaviour Detection Agent

6.2.1 Input

  • Full events[] array from AssessmentActivitySession.
  • criticalCount, warningCount, hasTabSwitch, hasPageClose, hasFullscreenExit flags.
  • screenSnapshot data for each event.

6.2.2 Decision Rules (Threshold-Based + AI)

ConditionDecisionAction
criticalCount = 0 AND warningCount <= 2cleanProceed to score evaluation
warningCount > 2 OR criticalCount = 1suspiciousAI flags but still evaluates score
criticalCount > 2 OR hasPageClose = true OR hasTabSwitch > 3rejectedAuto-fail regardless of score

6.2.3 AI Prompt Logic

  • Send events[] summary to Claude API.
  • Prompt: "Analyse this list of browser events during a proctored assessment. Determine if the behaviour is clean, suspicious, or indicates cheating. Return JSON: { decision, reason }".
  • AI decision overrides threshold if it detects a clear pattern not captured by simple counts.
  • Save activityDecision and activityRejectReason to AssessmentActivitySession.

6.3 Result Evaluator Agent

6.3.1 Input

  • score (percentage from Member 2).
  • passingPercentage from AssessmentStore.
  • activityDecision from Behaviour Agent.

6.3.2 Final Decision Logic

ScoreActivity DecisionFinal Outcome
score >= passingPercentagecleanAPPROVED — isVerifiedMentor = true
score >= passingPercentagesuspiciousAI reviews deeper — likely approved with note
score >= passingPercentagerejectedREJECTED — activity override. Score does not matter
score < passingPercentageanyREJECTED — score insufficient

6.3.3 Database Update

  • MentorProfile.verificationStatus = 'approved' or 'rejected'.
  • MentorProfile.isVerifiedMentor = true (only on approved).
  • MentorProfile.verifiedAt or rejectedAt = Date.now().
  • MentorProfile.rejectionReason = AI-generated string (on rejection).
  • Emit event to Email Pipeline.

6.3.4 Script Safety Rules

  • Idempotent: check if activityDecision already set. Skip if already processed.
  • Timeout: if AI API takes > 10 seconds, fall back to threshold-only decision.
  • Error: log failures with sessionId. Do not crash the process.

7. Low-Level Architecture — Email Notification (Member 3)

7.1 Trigger

Email is sent immediately after the AI Result Evaluator updates MentorProfile. Two emails are defined — one for each outcome.

7.2 Pass Email

FieldValue
Tomentor email from CommonUser
SubjectCongratulations! You are now a Verified Mentor 🎉
BodyVerification confirmed message. Platform dashboard access link. Next steps for setting up mentor profile.

7.3 Fail Email

FieldValue
Tomentor email from CommonUser
SubjectAssessment Result — Action Required
BodyResult reason (from rejectionReason). Retry available date (rejectedAt + 7 days). Improvement tips.

7.4 Implementation Notes

  • Use NodeMailer with SMTP or SendGrid HTTP API.
  • Store email templates as separate HTML files — do not hardcode in service.
  • Queue emails using a simple in-memory queue if needed — avoid blocking the AI callback.
  • Log email send status (success / failure) with mentorId for audit.
